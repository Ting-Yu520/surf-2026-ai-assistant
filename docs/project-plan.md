## 1. 项目摘要

### 中文

本项目研究一个问题：**当 AI 能"看懂"体育比赛后，如何让普通人也"看懂"AI 的分析结果？**

现有的 AI 体育分析系统（如 DeepMind 的 TacticAI）面向专业教练，输出的内容需要深厚的战术知识才能理解。一个完全不懂足球的人——潜在的新球迷——被排除在这个系统之外。

我们提出 **Generative HCI for Sports Analytics** 的概念：利用生成式 AI（LLM/VLM），将高维度的专业战术数据，转化为零知识门槛的、有情感吸引力的、可验证准确性的用户体验。系统工作流为：

```
📹 比赛视频 → 🔍 VLM 提取战术 JSON → 🧠 LLM 生成通俗叙事 → 👤 面向新手用户
```

本项目始于 SURF 2026（10 周，系统构建与原型验证），计划延伸为大四 FYP（用户研究与论文发表），最终目标是在 HCI（人机交互）或 Applied AI 领域发表高水平论文。

### English

This project addresses a critical gap in AI-powered sports analytics: **while AI systems like DeepMind's TacticAI can now understand and predict football tactics, their outputs remain inaccessible to non-expert audiences.** A person who knows nothing about football — a potential new fan — is excluded from engaging with AI-generated tactical insights.

We propose **Generative HCI for Sports Analytics**: using Generative AI (LLMs/VLMs) to translate high-dimensional professional tactical data into accessible, emotionally engaging, and verifiably accurate narratives for novice users. The system pipeline is:

```
📹 Match Video → 🔍 VLM extracts Tactical JSON → 🧠 LLM generates accessible narrative → 👤 Novice user
```

This project begins as a 10-week SURF program (system prototyping), extends into a Final Year Project (rigorous user studies), and targets publication at top-tier HCI venues (ACM CHI, IUI, DIS).

## 2. 研究空白分析

### 2.1 已有工作

| 研究 | 发表 | 做了什么 | 局限性 |
|------|------|---------|--------|
| **TacticAI** (Wang et al.) | *Nature Communications*, 2024 | 利用 GNN + 群等变 CNN 预测角球战术，AI 建议在 90% 情况下优于人类教练 | 面向专业教练；依赖利物浦独家 3D 追踪数据；不涉及"非专业受众" |
| **SportsBuddy** (Harvard/UMN) | *IEEE PacificVis*, 2025 | AI 驱动的多运动视频故事化工具，部署 150+ 用户 | 面向已有兴趣的球迷和教练；不研究"零基础"受众 |
| **Sportify** (Harvard/UMN) | *IEEE TVCG* | LLM + RAG 篮球战术视觉问答 | 假设用户有篮球基础知识 |
| **PXAI-Coach** (Stevens) | *ABC*, 2025 | 运动健康可解释 AI 仪表盘 | 面向运动员和教练，非"完全小白" |
| **Cricket XAI** (TU Delft) | 2025 | 按专业水平分层生成板球解释（初级→视觉、专家→统计） | 仅限于板球；未做大规模用户研究 |

### 2.2 关键空白

**无人在做"面向零认知受众的 AI 体育战术解释"。** 所有现有系统要么面向专业用户（教练、分析师），要么面向已有兴趣的球迷。但对于"对足球无感甚至抗拒"的人群——这是扩大体育受众的最大增量市场——没有任何 AI 系统考虑过他们。

### 2.3 学术机会

这个空白恰好落在 **HCI + Applied AI** 的交叉地带。该领域的顶会（ACM CHI、IUI、DIS）的核心关注点是"AI 系统如何改变人类认知和行为"，不要求发明新的底层算法。这完美契合 IMIS 学生的核心优势：Prompt Engineering、系统分析、用户需求理解。

## 3. 核心创新与方法论

### 3.1 学术贡献框架

本项目不是"做一个 AI 足球解说工具"，而是提出一个**可推广的框架**：

> **Generative HCI Pipeline for Domain Translation** — 将任意专业领域的高维结构化数据，通过 Prompt-Engineered LLM 管道，转化为特定受众可理解的、有情感吸引力的体验。

足球是本框架的第一个验证领域。其方法论可推广至医疗报告解读、法律文书简化、金融数据科普等场景。

### 3.2 系统架构

```
┌─────────────┐    ┌──────────────────┐    ┌─────────────────────┐    ┌──────────┐
│  比赛视频     │ → │  VLM 关键帧分析    │ → │  LLM Prompt 引擎     │ → │  新手用户  │
│ (公开转播)    │    │  · 球员识别       │    │  · System Prompt    │    │          │
│             │    │  · 位置提取       │    │  · 术语过滤         │    │  文本叙事  │
│             │    │  · 战术场景分类    │    │  · 生活类比生成      │    │  +        │
│             │    │  · → 结构化 JSON  │    │  · 防幻觉约束       │    │  画面高亮  │
└─────────────┘    └──────────────────┘    └─────────────────────┘    └──────────┘
    输入层              CV 层 (Phase 2)         核心算法层               输出层
```

### 3.3 Prompt Engineering 作为核心方法

本项目的核心 IP 不是模型本身，而是一套**结构化 Prompt 约束系统**：

| 约束维度 | 机制 | 示例 |
|---------|------|------|
| **角色约束** | System Prompt 定义"朋友式解说员"角色 | "你的观众这辈子可能第一次看足球" |
| **术语约束** | 黑名单 + 替代规则 | "越位"→"进攻球员跑得太快了，超过了规则允许的位置" |
| **类比约束** | 强制要求每个战术场景对应一条生活类比 | "这就像你在早高峰地铁站..." |
| **数据约束** | 只能基于提供的 JSON，禁止编造 | 不确定处必须标注 [推测] |
| **格式约束** | 自然段落、禁止 markdown、禁止 emoji 标题 | 像朋友聊天，不像教科书 |

### 3.4 验证方法

**核心假设：** AI 生成的通俗叙事，相比传统解说，能显著降低零基础受众的认知负荷。

**验证手段（FYP 阶段）：**
- **实验设计：** 被试间设计（Between-subjects），随机分配
- **自变量：** 解说类型（传统 vs AI 生成）
- **因变量：** 认知负荷（NASA-TLX）、理解准确度（自研测试）、观看兴趣（Likert 量表）
- **样本量：** ≥30 人/组（通过 G*Power 先行计算）
- **统计方法：** 独立样本 t 检验 / Mann-Whitney U 检验
- **定性补充：** 半结构化访谈 + 主题分析

### 5.2 每周详细产出

| 周 | 核心任务 | 可交付产出 |
|----|---------|-----------|
| **W1** | 文献调研：TacticAI + SportsBuddy + HCI 相关论文 10+ 篇精读 | 文献综述笔记 |

| **W8** | UI 完善：视频/图片上传、多场景对比、历史记录 | 更新版 Streamlit App |
| **W9** | 预实验：5-10 人小型测试，收集反馈 | 预实验数据 + 反馈汇总 |
| **W10** | 文档完善：技术文档、用户手册、FYP 开题材料准备 | 完整项目文档 |

## 8. 风险评估与应对

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| **审稿认为 novelty 不足** | 中 | 高 | 论文叙事定位在"方法论框架"层面，不是"应用工具"；强调跨领域可推广性 |
| **VLM 提取 JSON 不稳定** | 中 | 中 | Phase 1 允许人工辅助修正；Phase 2 用 Few-shot Prompting + Schema 约束提高准确率 |
| **用户研究被试不足** | 低 | 高 | 尽早发招募广告；提供报酬；利用学校社交网络 |
| **统计结果不显著** | 低-中 | 高 | 预实验做功效分析确保样本量；如效应量小，调整为"AI ≠ 更差，但有独特优势" |
| **CHI 被拒** | 中-高 | 中 | 准备了 IUI/DIS 备选；被拒后根据审稿意见修改重投 |
| **时间不够** | 低 | 低 | SURF 只负责技术，FYP 有一整年做用户研究 |
